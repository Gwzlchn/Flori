# 14 · 注释与文档风格

> 权威版。CLAUDE.md「注释与文档风格」速查节是本文的摘要,冲突以本文为准。
> 审计底档与改造清单在 `.local/processing/注释与文档准则.txt`(不入 git)。

## 0. 总则

注释解释代码"为什么这么做",文档解释系统"是什么、怎么组织"。
能一句直陈说清的,不加装饰符号、不套括号、不讲已经不存在的历史。

全仓审计结论:注释的 why 密度高、质量不差,主要问题是装饰与噪声。
所以本准则的重点是减法——去装饰、拆括号、把版本与追踪信息迁出注释,而不是补更多注释。

适用范围:代码注释与文档正文。运行时输出的 UI 文案不受约束——CI step summary、
前端界面文案、给人看的日志,按各自 UI 语境判断。

## 1. 硬性禁止 / 必须速查表

| 禁止 | 改成 |
|---|---|
| box-drawing 分隔线 `# ── x ──`、`════`、`=====` 当章节头 | 普通 `# 段名`,或靠空行、函数/文件拆分导航 |
| ★ ● ✅ ⚠️ ↑ · 等符号当强调、图例、项目符号 | 直陈句;列举用 `1. 2.` 或 `-` |
| ① ② ③ 圈码当步骤序号 | 普通 `1. 2. 3.` |
| 【】 包普通词当强调 | 去掉方括号,直接写这个词 |
| 「」『』 当强调引号 | 直陈;「」只保留给引用真实 UI 文案这一种用途 |
| 全角标点 `（）：，；` | 半角 `() : , ;`(中文句号 `。` 可保留,全文一致) |
| 一句话套两层以上括号补充 | 拆成两三个直陈短句 |
| 注释里写版本号、阶段码、审计号、issue 号、commit sha、日期 | 只写当前行为;沿革进 git commit 或 worklog |
| 解释"旧的/此前/原来/不再/已删"的过时方案 | 只描述现状;历史进 git |
| 复述文档已有的领域规则、状态机、schema、契约字段 | 一行指向 docs,见 §5 |
| 复述紧邻代码的 docstring / 行尾注释 | 删掉,或补一条真正的 why/边界 |
| 注释掉的死代码 | 直接删,git 里能找回 |

| 必须 | 说明 |
|---|---|
| 注释讲 why、坑、不变量、边界返回 | 讲代码看不出来的信息,不翻译代码 |
| 注释与 docstring 用半角标点 | 逗号/括号/冒号/分号一律半角 |
| 反直觉设计指向 docs/ADR | 写"见 docs/xx §y"或"见 ADR-00NN",不粘贴理由正文 |
| 单一事实源用指路代替复制 | 如"格式见 api/services/radar.py"、"已统一到 utils/format.ts" |

## 2. 装饰字符:去掉不损信息

### 2.1 分隔线

分隔线不带信息,靠函数名和空行就能导航。

```
坏:  # ── Lua 脚本 ──
好:  # Lua 脚本
```

一个类里需要五六条分隔线来分组,通常说明该拆类/拆文件,拆分比分隔线好。

### 2.2 强调符号与强调括号

```
坏:  # ★用 replace 注入(prompt 含字面 {},不可 str.format)。
好:  # 用 replace 注入:prompt 含字面 {},不能用 str.format。

坏:  # 序列化对【单一共享连接】的写访问。
好:  # 序列化对单一共享连接的写访问。

坏:  # 区分「刻意免费」与「配置漂移漏命中」。
好:  # 用来区分刻意免费和配置漂移漏命中。
```

「」唯一合法用途:引用真实存在的 UI 文案,如按钮文案 `「重跑该步」`。
引用代码里的字面名(CI step 名、配置键)用半角引号,如 `"稳定 pip 缓存" 步`。

### 2.3 全角标点

```
坏:  """SYSTEM：AI Provider 调用失败（网络/服务端 5xx），退避重试。"""
好:  """SYSTEM: AI Provider 调用失败(网络/服务端 5xx),退避重试。"""
```

## 3. 不要大量用括号

判据:一句话里出现第二个补充括号,或括号里再嵌因果链,就拆句。

```
坏:  # token 总数最大的键(主力模型);都拿不到 → "subscription"(订阅路径占位,不迁移历史数据)。
好:  # 取 token 总数最大的键,即主力模型。都拿不到时用 "subscription" 占位,这条路径不迁移历史数据。
```

`→` 是约定俗成的"映射到 / 导致"简写,数据流描述里可用,但不拼三段以上的链条,那种拆成句子。

## 4. 版本号、changelog、追踪码不进注释

注释描述现在的行为。哪个版本引入、哪次审计发现、哪个 commit 修的,属于 git log 和 worklog,
读代码的人查不到也不需要。禁止入注释:版本号、阶段码(P2b/Phase2/C1)、审计号(R-M9/I-H4)、
commit sha、日期、没写文档名的裸章节号(`§8.6`)。

```
坏:  # 清单步重试计数(rerun 用):否则重跑曾耗尽重试的步骤会零重试预算(审计 I-H4)。
好:  # 清单步重试计数,供 rerun 用:否则重跑曾耗尽重试预算的步骤会零重试预算。
```

## 5. 不解释过时设计,不复述文档

### 5.1 过时设计

注释讲一个已删掉的旧方案是负信息:读者拿着注释去代码里找,找不到。
凡"旧的 / 此前 / 原来 / 不再 / 已删"起头的整段,只留现状陈述。
已修复的旧 bug 同理:保留"为什么这么写"的不变量,删掉"原来错在哪"的复盘。

```
坏:  # 注:旧 WORKER_POOLS(type→默认池映射)已删——能力统一用 --pools 显式表达。
好:  # 能力用 --pools 显式表达,路由按 pool 走。
```

### 5.2 文档解释代码,代码指向文档

以下主题由 docs 单一权威承载,代码注释只指向、不复述:

- 领域模型、状态机、DB schema → `docs/02-domain-model.md`
- 对外接口契约(API/WebSocket/Redis 键/文件 Schema/错误码/MCP) → `docs/03-contracts.md`
- 架构、部署拓扑、资源池、依赖规则 → `docs/01-architecture.md`;安全边界 → `docs/07-security.md`
- 架构决策的为什么 → `docs/adr/*`
- 幂等/指纹/级联失效 → `docs/04-module-design/step-base.md`;Prompt 组装 → `docs/06-prompt-engineering.md`

反向同样成立:具体实现细节、配置文件的逐字段当前值留在代码/配置,docs 只给结构与示例。
前端不把后端 JSON 契约抄进注释,指向后端 services 文件:

```
坏:  // 形状(后端 api/services/radar.py): { rising_concepts:[{term,recent,...}], ... }
好:  // 返回形状见 api/services/radar.py,以后端为准。
```

### 5.3 引用文档的确切写法

章节号用 `§` 加数字,ADR 用 `ADR-00NN`,必须带文档名:

```
# 见 docs/03-contracts.md §4.8
# 状态机见 docs/02-domain-model.md §2.2
# 理由见 ADR-0011
# fail-closed,见 docs/07-security.md
```

## 6. 死代码与 TODO

注释掉的可执行代码一律删除。例外:配置文件里的可覆盖参数模板,块首写清"按需取消注释覆盖"。

TODO 保持低量,格式 `# TODO: 一句话说清要做什么 + 触发条件`。
长期规划进 ROADMAP 或 worklog。

## 7. docstring 规范

不用 Google/NumPy 形式化 Args/Returns 段,参数类型靠 type hints 表达。docstring 讲意图和约束。

- 模块级:三引号,首行一句中文点题;复杂模块下接多行,讲数据流、契约、坑。
- 函数级:做什么 + 为什么/坑/不变量 + 边界返回(如 `失败返回 None`、`缺/坏 → {}`)。
- 函数名加签名已自明且没有坑的,省略 docstring;不写 `"""检测是否有 GPU。"""` 这种复述式。
- FastAPI 路由的 docstring 会进 OpenAPI 描述,属对外行为:按风格改写,不随手删。

## 8. 前端 Vue/TS 差异

- 不用 JSDoc/TSDoc。文件级说明放 `<script setup>` 顶部一段 `//` 块:职责 + 单一事实源 + why。
- interface 字段用行尾 `//` 标注含义和 null 条件。
- `●` 指代 UI 圆点时写"圆点角标",不写符号。

## 9. 什么是好注释

保留并鼓励:踩过的坑与规避、并发不变量、安全约束、魔法常量的实测依据、选型原因与失败症状。

判断标准:删掉这条注释,后人会不会踩坑或误改?会,就留;
不会(只是翻译代码/复述文档/讲历史),就删。
